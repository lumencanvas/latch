## Fixed this audit

Resource leaks — all in the bug class CLAUDE.md flags (module-level per-node
state without a wired cleanup path):

- **mqtt / websocket / http executors leaked live connections.**
  `gcMqttState` / `gcWebSocketState` / `gcHttpState` and their `disposeAll*`
  were defined and exported from `engine/executors/index.ts` but **never
  imported or called by `ExecutionEngine`** — these executors override the
  legacy connectivity ones for `'mqtt'`/`'websocket'`/`'http-request'`, so
  `gcConnectivityState` (which only covers OSC/Serial/MIDI/BLE) never touched
  their state. Result: `WebSocket`s from mqtt/websocket nodes were never closed
  on node deletion *or* on stop/reset. Wired all three into `updateGraph`'s GC
  path and `stop()`'s teardown.
- **`utility` / `messaging` executors had no per-node GC.** Both had a
  `disposeXNode(id)` but no `gc*State(validNodeIds)`, so deleting an individual
  latch/counter/debounce/throttle/sampleHold/changed or send/receive node leaked
  its entry until a full stop. Added `gcUtilityState` / `gcMessagingState` and
  wired them into the per-node GC path.
- **VLA (SmolVLM) model leaked GPU/WASM on unload.** `ai.worker.ts` caches
  vision-language models as a `{ processor, model, vlm: true }` wrapper with no
  top-level `dispose()`, but `handleUnload` / `handleDispose` / `handleClearCache`
  only called `pipe.dispose()` — so the guard was false and `model.dispose()`
  never ran. Added a `disposePipe()` helper that disposes the wrapper's
  `model` + `processor`, used on all three paths. (This is session-introduced
  code — the VLA node was added this branch.)
- **`AudioManager.resume()` left the "Enable Audio" UI stale.** It mutated
  `_needsUserGesture` but, unlike `unlock()` / `handleContextStateChange()`,
  never called `notifyListeners()`, so the StatusBar affordance didn't refresh
  after a resume-path recovery. One-line fix.

UI / theme (earlier this session, for context):
- Reverted the unwanted AI teal recolor — AI keeps its purple sub-brand.
- Redesigned the per-protocol connection forms (2-col grid, mono labels, removed
  the duplicated auto-connect/auto-reconnect fields), restyled the Links sidebar

## Outstanding — prioritized (NOT fixed; recommend follow-ups)

### HIGH
- **`BleAdapter` notification listener never removed. ✅ FIXED 2026-06-19.**
  `notificationHandlers` stored a *different* closure than the one added as
  `characteristicvaluechanged`, so the real listener could never be removed and
  stacked (each retaining `this`) on every reconnect. Fixed: store the *same*
  `handler` that's attached; remove it in `unsubscribeFromNotifications()`, a new
  `detachNotificationListeners()` helper (used by `doDisconnect()` + `dispose()`),
  and drop the previous listener when re-subscribing the same characteristic.
  Covered by `tests/unit/services/connections/BleAdapter.test.ts`.
- **`--radius-xs` is undefined and used at ~60 live call sites.** Not in
  `tokens.css` (scale is none/sm/default/md/lg/full) → silently falls back to the
  initial `0`. In a brutalist theme sharp corners are arguably fine, so this is
  "decide intent" rather than purely cosmetic: either define `--radius-xs`
  (e.g. 1–2px) or repoint these to `--radius-none` / `--radius-sm`. Widespread:
  PropertiesPanel, AppSidebar, AIModelManagerModal, AssetPicker/Browser/Card,
  several input/timing node components.

### MED
- **Undefined numbered error/success/warning + `-bg`/`-light`/`-ink` color
  tokens.** Only the base `--color-error/success/warning` exist; refs like
  `--color-success-100/700`, `--color-error-50/200/500/600`, `--color-warning-400`,
  `--color-error-light`, `--color-warning-ink` fall back (some have inline
  fallbacks, some don't). Sites: `TemplateSelect.vue`, `HttpTemplateEditor.vue`,
  `AssetPickerControl.vue`, `AssetCard.vue`, `CodeEditorModal.vue`,
  `DebugPanel.vue`, `StatusBar.vue`. Add the scales to `tokens.css` or repoint.
- **Other undefined tokens:** `CodeEditorModal.vue:344` `--color-neutral-850`;
  `HttpTemplateEditor.vue:566` `--shadow-xl`; `ProtocolSelector.vue:177`
  `--space-0-5`; `NodeConnectionStatus.vue:108` `--space-1-5`.
- **`subflow.ts` contexts never cleaned.** `subflowContexts`
  (`engine/executors/subflow.ts:15`) has `clearSubflowContext` /
  `clearAllSubflowContexts` defined but called from nowhere — app-lifetime leak.
  Wire `clearAllSubflowContexts()` into `ExecutionEngine.stop()` and a per-node
  clear into the GC path.
- **`TextureBridge` caches not reachable from `gc()`.**
  `services/visual/TextureBridge.ts` `gc()` only cleans `displaySprites`;
  `pixiToThreeCache` / `threeToPixiCache` only free via explicit dispose. Also
  `gc()`'s `id.split('_')[1]` assumes a `node_<id>_…` format the creator doesn't
  enforce. Make node removal free the cached texture pair.
- **`AIInference` worker `onerror` doesn't reject pending requests.** A worker
  crash only `console.error`s; in-flight requests hang until the 5-minute
  timeout. Reject all pending on `onerror`; add a `default` branch for unknown
  message types.
- **`ai.ts` `disposedNodes` guard may be ineffective (corrected).** `disposedNodes`
  *is* read — by `isNodeDisposed()` (ai.ts:27). The real question is whether
  `isNodeDisposed()` is actually called at the in-flight async callback sites
  before they `setCached(...)`; if not, a node removed mid-inference re-populates
  its cache (self-corrects next gc tick). Confirm the guard is applied at the
  callback sites, or remove it. (Downgraded from the first pass's "dead code"
  claim, which was imprecise.)
- **Gradients vs flat-fill theme (non-AI).** `ConnectionManagerModal.vue:211`
  (teal modal-header gradient — introduced this session), `ClaspBridgeDownload.vue`,
  `MonitorNode.vue`, trigger/xy-pad node button gradients.
- **Purple outside AI.** `EQEditor.vue:22`, `ParametricEqNode.vue`,
  `SynthNode.vue`, `ControlPanelView.vue` use `#a855f7`/`#8b5cf6` accents on
  non-AI audio controls; `stores/nodes.ts` colors several non-AI categories
  purple/violet. Repoint to `--color-primary-*` if theme unity is wanted.
- **`createTextureFromWebGL` returns untracked `DataTexture`** in both
  `ThreeRenderer.ts:~600` and `ThreeShaderRenderer.ts:~780` — caller-owned, no
  dispose contract. `ThreeRenderer.disposeObject` also doesn't dispose material
  textures (`.map` etc.) or light shadow maps.

### LOW
- **`ThreeShaderRenderer.materials` is vestigial dead code (corrected — NOT a
  leak).** The first pass flagged this HIGH ("compiled `ShaderMaterial`s never
  disposed"). Re-verified false: `compileShader()` *returns* the material to its
  caller, `engine/executors/visual.ts`, which owns it in `compiledShaderMaterials`
  and disposes it in `disposeVisualNode` / `gcVisualState` / `disposeAllVisualNodes`
  (all wired into the engine). So materials are disposed correctly; only the
  renderer's own `this.materials` Map is never populated, making `disposeNode()`'s
  material branch dead code. Remove the dead Map/branch for clarity.
- **clasp `autoConnect` fallback divergence.** The editor forces `autoConnect:false`
  for new connections, but `executors/clasp.ts` still falls back to
  `autoConnect ?? true` when the key is absent — a node-driven / legacy config
  with no key still auto-connects, against the "off by default" intent. Align the
  fallback to `false`.
- **Palette double-add.** `AppSidebar.vue` palette items bind both `@dragstart`
  and `@click="requestNodeAdd"`; an aborted drag can emit a trailing click →
  spurious node add. Add a drag-occurred guard.
- **Hardcoded hex duplicating tokens:** `stores/nodes.ts` (#2AAB8A, #8B5CF6),
  `ControlPanelView.vue` (#00d4aa ≈ but ≠ `--color-primary-500`).
- **Ad-hoc soft shadows** on overlays/nodes that could reference
  `--shadow-elevated` / `--shadow-subtle` instead of inline rgba.

## Pass 2 — security + async robustness (2026-06-16)
Two further read-only passes on dimensions the first audit didn't cover. Items
marked **[verified]** were re-traced personally; the rest are from the audit
agents and should be confirmed before fixing. Caveat used throughout: the
standalone `mqtt.ts`/`websocket.ts`/`http.ts` executors **override** the legacy
`connectivity.ts` ones for those node types, so some `connectivity.ts` findings
are on dead code paths (its OSC/Serial/MIDI/BLE executors are still live).

### Security — untrusted code execution is the headline risk
- **[verified] HIGH — the "sandbox" is not a sandbox.** `engine/executors/code.ts:93`
  (Function node) and `:183` (Expression node) run user code via
  `new Function('ctx', 'with (ctx) { … }')`. `with(ctx)` only *adds* ctx to the
  scope chain; it removes nothing, so user code reaches `window`, `document`,
  `fetch`, `WebSocket`, `localStorage`/IndexedDB, the Pinia stores, and (Electron)
  the `window.electronAPI` preload bridge. The UI/header calls this "sandboxed
  execution" (`registry/code/_function/definition.ts`), which is false and
  misleading. Same non-sandbox in the custom-node compiler
  (`services/customNodes/compiler.ts:78,133`).
- **HIGH — flow import does no validation.** `stores/flows.ts` `importFlow`/
  `importFlows` JSON.parse → fields copied verbatim; no schema/allow-list/scrub of
  `code`/`expression`/`url`/`autoConnect`. A shared `.latch` flow can therefore
  carry attacker JS (runs on Play — one-click, not zero-click) and auto-connecting
  nodes pointed at attacker servers. The dangerous case (code arriving in someone
  else's flow) is indistinguishable from the safe case (you typing your own code)
  because import separates neither.
- **HIGH (Electron) — `setWindowOpenHandler` opens any URL/scheme.**
  `src/main/index.ts:33` hands every `window.open(...)` to `shell.openExternal`
  with no `http(s)`-only allow-list; reachable from user node code. Combined with
  the exposed `electronAPI` (`preload.ts`: `openExternal`/`openPath`/asset
  read/write/delete), a malicious flow on desktop can launch arbitrary URLs/paths
  and touch asset files. (Renderer is otherwise hardened: `nodeIntegration:false`,
  `contextIsolation:true`.)
- **MED — outbound targets unrestricted.** HTTP-request node (`connectivity.ts`
  fetch) and WebSocket/MQTT `autoConnect` take `url` straight from controls with no
  host/scheme block-list (SSRF to internal/`169.254.169.254`; exfil/beacon).
- **LOW — no CSP** (`index.html`, no main-process header); a tight `connect-src`
  would blunt exfil. **Positives:** no `v-html`/`innerHTML`/`document.write`
  anywhere (XSS sink surface genuinely low); `CredentialStore` keeps secrets out of
  flow JSON and uses Electron `safeStorage`; tokens aren't logged.
- **Framing:** running user code is by design — the fix is to (a) stop calling it
  sandboxed, (b) actually bound it (Worker with a curated API, or at least a
  global-shadowing preamble), and (c) gate code-bearing / auto-connecting imported
  flows behind an explicit trust prompt.

### Async robustness — per-frame storms + wedged states
- **[verified] HIGH — imageLoader asset-failure storm. ✅ FIXED 2026-06-19.**
  `executors/visual.ts:1378`: the asset-not-found branch set `error`/`loading=false`
  but never set `state.loadedUrl`, so the load gate stayed true and re-fired
  `getAssetUrl` every frame for any missing `assetId`. Fixed: the not-found *and*
  catch branches now latch `state.loadedUrl = assetUrlKey` (a Trigger still forces a
  retry). The URL branch already latched upfront, so it was never affected.
- **HIGH — webcam-snapshot re-requests `getUserMedia` every frame after a denial**
  (`visual.ts:~1602`); no permanent-fail latch. **✅ FIXED 2026-06-19** — added a
  `failed` latch that skips the retry until the requested device/resolution changes
  (cleared on a successful init).
- **HIGH — http-request node floods** when `trigger` is held true
  (`executors/http.ts:78`): the `:loading` flag is written but never read to gate a
  new request → a new `fetch` per frame, responses interleave. **✅ FIXED 2026-06-19** —
  now fires on the rising edge of `trigger` only and reads the in-flight `:loading`
  flag to gate, so a held trigger fires exactly one request. Covered by unit tests.
- **HIGH — per-frame connect storm** for a persistently-failing mqtt/ws/http/BLE
  connection (`ensureConnected` in each; BLE `connectivity.ts`): once the adapter
  settles to `disconnected`/`error`, the executor issues a real connect ~60×/s; no
  executor-level backoff. BLE can overlap connects (gates on `!connected` while
  status is `connecting`).
- **HIGH — ConnectionManager `disconnect` masks errors → wedged toggle.**
  `services/connections/ConnectionManager.ts:237`: catch only logs; `statusCache`
  stays `connected` on a throwing `adapter.disconnect()`, so the store toggle reads
  stale state. Feeds `updateConnection`'s `disconnect().then(connect)` which can
  resurrect a just-removed connection / reconnect over a live socket.
- **MED — clasp `disconnect()` mutates shared `connection.config.autoReconnect=false`**
  (`executors/clasp.ts:228`) for ALL nodes sharing that connectionId; plus
  `error`-state-with-no-retry wedge, and a connect `Promise.race` timeout that
  leaks the late-resolving client. Serial/BLE duplicate device-picker dialogs
  (connect awaited before map insertion).
- **LOW — confirmed:** `ai.ts` `disposedNodes` is written but the async inference
  callbacks never call `isNodeDisposed` before `setCached` (state resurrection
  until next gc sweep) — matches the corrected first-pass note.
- **Notably robust:** `ExecutionEngine`'s async lifecycle (`loopToken`,
  `acceptAsyncResults` + `nodeById.has`, per-node try/catch), `WebLLMService`
  (genToken supersession, load coalescing, interrupt-before-dispose), and the
  `AIInference` worker request map (self-clearing timeouts, dispose rejects all
  pending) — no issues found.
