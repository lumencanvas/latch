## TIER 1 — Security (HIGH; carried from 2026-06-16, all still open)

The headline is a single chain: **a shared `.latch` flow is one click (Play) from arbitrary
code execution with full Electron power.** Running user code is by design — the fix is to
*bound* it, *relabel* it, and *gate* untrusted imports.

1. **The "sandbox" is not a sandbox.** `engine/executors/code.ts:93,183` (Function/Expression)
   and `services/customNodes/compiler.ts:78` run user code via `new Function('ctx','with(ctx){…}')`.
   `with` only *adds* to scope — user code reaches `window`/`document`/`fetch`/`WebSocket`/
   `localStorage`/Pinia/`window.electronAPI`. The UI still labels it "sandboxed"
   (`registry/code/_function/definition.ts`) — **false**. Fix: Worker isolation or a
   global-shadowing preamble; relabel.
2. **Flow import does zero validation** (`stores/flows.ts` `importFlow`/`importFlows`): node
   `data` (incl. `code`/`expression`/`url`/`autoConnect`) copied verbatim; runs on Play. Fix:
   trust-prompt on import (flag code-bearing / auto-connecting nodes), or a safe-mode that
   disables code exec; consider regenerating node ids on import.
3. **`setWindowOpenHandler` opens any scheme** (`src/main/index.ts:33-34`): `shell.openExternal(details.url)`
   with no `http(s)` allow-list; reachable from node code via `window.open`. Also the
   `shell:openExternal` / `shell:openPath` IPC handlers (`index.ts:138-146`) call shell with
   renderer-supplied url/path unchecked. Fix: `http(s)`-only allow-list.
4. **Electron IPC path traversal** (`src/main/index.ts` `customNodes:readDefinition`/`readExecutor`/
   `scanDirectory`, ~:191-208, and any `assets:*` file handlers): `join(basePath, userInput)`
   with no containment — `packageName='../../…'` escapes the base dir (read/write/delete).
   Fix: `path.resolve` + assert the result stays under the base dir. **[verified]**
5. **No CSP + unrestricted outbound / SSRF** (`index.html`; http/ws/mqtt executors take `url`
   from controls): no `connect-src` limit, no host block-list (internal IPs / `169.254.169.254`).
   Fix: add a CSP `connect-src`, and a host/scheme allow-list on outbound connections.

Positives: `nodeIntegration:false`, `contextIsolation:true`, no `v-html`/`innerHTML`,
`CredentialStore` uses Electron `safeStorage`.

## TIER 2 — HIGH robustness / leaks (still open)

6. **Per-frame connect storm** (`engine/executors/{mqtt,websocket,http}.ts` `ensureConnected`
   + BLE path in `connectivity.ts`): once an adapter settles to disconnected/error, the
   executor calls `connectionsStore.connect()` ~60×/s with no per-attempt throttle. The
   adapter's own backoff only covers *unexpected* disconnects. Fix: executor-level
   `lastAttempt` + min-retry-interval gate (same shape as the http-flood fix). *The #1
   remaining per-frame storm.*
7. **`ConnectionManager.disconnect` masks errors → wedged toggle** (`services/connections/ConnectionManager.ts:~237`):
   the catch only logs; `statusCache` stays `connected` on a throwing `adapter.disconnect()`,
   and `updateConnection`'s `disconnect().then(connect)` can reconnect over a live socket. Fix:
   set `error`/`disconnected` + emit in the catch, and re-throw. Small.
8. **`AIInference` worker `onerror` doesn't reject pending** (`services/ai/AIInference.ts:269`):
   a worker crash only `console.error`s; in-flight requests hang to the 5-min timeout. The
   dispose path (`:752`) rejects pending, but a *crash* doesn't. Fix: reject all
   `_pendingRequests` in `onerror`; add a `default` branch in `ai.worker.ts` that replies
   `success:false`. **[verified]**
9. **History store: unbounded memory + no cleanup on flow delete** (`stores/history.ts`): deep
   JSON snapshots per change, 20/flow × N flows, and `deleteFlow` never clears the
   undo/redo stacks. Fix: clear history on flow deletion; consider a global size budget.
10. **Assets: blob URLs not revoked + orphaned node refs on delete** (`stores/assets.ts`
    `deleteAsset`): the blob URL is never `revokeObjectURL`'d and referencing nodes keep a
    dead `assetId`. Fix: revoke on delete; warn/clear referencing nodes.

## TIER 3 — MED (open)

11. **GPU texture-ownership leaks** (carried from 2026-06-16, **still open** — confirmed):
    - `createTextureFromWebGL` returns an untracked `DataTexture`, allocated **every frame** a
      shader receives a WebGL-texture input (`ThreeRenderer.ts`/`ThreeShaderRenderer.ts` +
      `executors/visual.ts:~540`). No pool/dispose.
    - `ThreeRenderer.disposeObject` disposes geometry+material but **not** material `.map`/
      `normalMap`/etc. or **light shadow maps** → GPU leak on 3D node removal.
    - `TextureBridge.gc()` only frees `displaySprites` (the `pixiToThreeCache`/`threeToPixiCache`
      leak) and uses a fragile `id.split('_')[1]`.
    - Render-target pool (`getOrCreateRenderTarget`) keyed by node+size with no eviction.
12. **AI `isNodeDisposed()` guard is dead** (`engine/executors/ai.ts:26`): defined/exported but
    **never called** at the async `setCached` callback sites, so a node deleted mid-inference
    resurrects its cache (~100ms window; self-heals next gc). Fix: guard each async callback.
    **[verified — only the definition references it]**
13. **clasp**: `disconnect()` mutates the **shared** `connection.config.autoReconnect=false`
    for all nodes on that connectionId (`executors/clasp.ts:~228`); the connect `Promise.race`
    timeout leaks a late-resolving client; `autoConnect ?? true` fallback diverges from
    "off by default."
14. **MediaPipe video/camera stream not stopped** on node removal (`services/ai/MediaPipeService.ts`):
    landmarker `close()`d but the `HTMLVideoElement`/MediaStream keeps running.
15. **STT `AudioBufferService` has no `dispose()`** (`executors/ai.ts`): disconnected but
    internal buffers/state accumulate across create/destroy.
16. **`clearModelCache` doesn't cancel in-flight loads** (`AIInference.ts`): a pending load can
    re-populate `_loadedModels` after the clear.
17. **BLE duplicate device-picker** on a connect race (`connectivity.ts` bleDevice): adapter
    map-set before the await completes; no creating-guard.
18. **Audio analyser per-frame FFT cost** (`executors/audio.ts`): 256-pt FFT + Meter every
    frame per analyser, no skip/cache (perf, not a leak).

Five parallel adversarial passes: (1) verify this cycle's 6 fixes, (2) engine + executor

## A. Verification of the 6 landed fixes — 3 solid, 3 need a follow-up correction

- **#7 disconnect, #8 AIInference onerror, #9 deleteFlow history — SOLID.** Re-read and
  confirmed correct, complete, no regressions. (onerror caveat: a crashed worker is not
  recreated — pre-existing recovery limit, not introduced here.)
- **CORRECTION to #3 (Electron IPC) — `shell:openPath` + `shell:showItemInFolder` were NOT
  guarded** (`src/main/index.ts:161-167`). They take a renderer-supplied path raw. *Mitigating:*
  both are **unbridged** — no `preload` exposure and no renderer caller — so they're latent dead
  surface, not a live hole. Fix: guard with a path check or remove the unused handlers.
  Also: `containedPath` is **lexical-only** (no `realpath`), so an in-dir symlink could still
  escape — low (attacker must first plant a symlink). `assets:setBasePath` lets the renderer
  point the asset root anywhere (by design).
- **CORRECTION to #1 (code restriction) — `services/customNodes/compiler.ts:78` runs custom-node
  executor source with NO preamble.** Defensible (installed-on-disk files are a higher trust tier
  than flow-embedded Function nodes) but undocumented — add a comment making the asymmetry
  intentional, or extend the preamble. The Function/Expression relabel itself is honest.
- **CORRECTION to #6 (connect throttle) — the `lastConnectAttempt` Map keys are never cleaned**
  in any `disposeX`/`gcXState` path (mqtt/ws/http). One small int per connectionId, bounded by
  distinct-connection count — a minor leak that contradicts the "MUST register cleanup" rule.
  BLE was correctly left unthrottled (it's connect-edge + user-dialog gated, never a per-frame
  redial).

## B. NEW findings (not previously documented)

**Stores / UI persistence**
- **HIGH — `exposeControl` never persists** (`stores/ui.ts:457-458`): pushes the control and
  returns with no `saveToStorage(STORAGE_KEY_EXPOSED_CONTROLS, …)` — every sibling mutation saves.
  A freshly exposed control is **lost on reload** until some other control edit flushes the array.
  One-line fix.
- **MED — `controlPanelLayout` orphaned on node delete** (`stores/ui.ts`): `removeControlLayout`
  exists but is called nowhere; node-delete runs `cleanupExposedControls` but never prunes the
  persisted layout → unbounded growth + ghost rectangles bump new controls in collision math.
  Add a `cleanupControlLayout(existingNodeIds)` sibling.
- **LOW/MED — `connections` store mutates the statuses Map in place** (`stores/connections.ts:78,85`):
  `connection-added`/`connection-removed` use raw `.set`/`.delete`; the `status-change` path
  correctly clones. A new connection can render with no/stale status until the next clone. Use the
  clone-on-write pattern everywhere.
- **LOW — import reuses inner node ids verbatim** (`stores/flows.ts:408-436,920-976`): flow id is
  regenerated, node/edge ids are not — a double-import or cross-flow id collision can confuse
  node-keyed executor state. (Distinct from the known #2 *trust* angle.) Remap ids on import like
  `duplicateFlow` already does.

**Engine / executors**
- **HIGH — Oscilloscope/Equalizer leak their Tone analyser node** (`executors/index.ts:1150-1165,
  1197-1214`): cleanup calls `disconnect` but never `.dispose()` on the `Tone.Waveform`/`Tone.FFT`
  (every other Tone node disposes). Underlying Web Audio AnalyserNode leaks per add/remove + restart.
- **HIGH — Equalizer "audio disappeared" branch leaks the FFT** (`executors/index.ts:1086-1090`):
  drops a live `Tone.FFT` to `null` with no disconnect/dispose — triggers on a normal runtime
  rewire, worse than the teardown leak. Oscilloscope signal-mode path (`:1028-1033`) has the
  mirror gap.
- **LOW — `delayExecutor` has no input dedupe** (`executors/index.ts:734-736`): enqueues every
  frame the input is defined (~60/s for a steady upstream); bounded in steady state but allocates
  per frame. Same class as the counter/sample-hold dedupe already fixed.

**Visual / GPU — #11 confirmed, severities refined**
- **HIGH, per-frame — `createTextureFromWebGL` allocates an untracked `DataTexture` every frame**
  on the *raw-WebGLTexture → Shader node* path only (`executors/visual.ts:540,562` →
  `ThreeShaderRenderer.ts:762-785`): ~1 MB CPU + 1 MB VRAM/frame, plus a `readPixels` GPU→CPU
  stall, none disposed. The priority #11 fix: cache+reuse one DataTexture per (node,uniform),
  dispose in `disposeNode` — **self-owned, no shared-texture risk.** (The `ThreeRenderer.ts:584-604`
  copy is dead/unreferenced — delete so it can't be rewired.)
- **MED, per-removal — material maps + light shadow maps never disposed** (`ThreeRenderer.ts:609-619`;
  `3d.ts:142-160,126-137`): `Material.dispose()` does not free its textures; GLTF maps + shadow
  `WebGLRenderTarget`s leak on 3D-node removal. ⚠️ **Shared-ownership hazard:** material `.map` may
  point at a node-cached `videoTextures`/`dataTextures`/`canvasTextures` entry — disposing it
  blindly can blacken another node. Safe split: **dispose shadow maps unconditionally** (always
  self-owned); handle material maps only at the owning cache layer, never via a blind traverse.
- **LOW (inert) — `TextureBridge.gc()` partial + `id.split('_')[1]`** (`TextureBridge.ts:281-291`):
  real but currently dead (the threeToPixi/pixiToThree caches have zero callers). Fix or delete
  before the Pixi↔Three interop is wired; if revived, `destroy(false)` to keep the shared GL texture.
- **MED, per-resize — render-target pool has no eviction** (`ThreeShaderRenderer.ts:707-726`,
  `ThreeRenderer.ts:211-240`): a size-animated node mints one RT per distinct size (all freed on
  removal). Evict stale-size targets for the node at the top of `getOrCreateRenderTarget`.

**Audio / AI / connectivity — #13–#17 reassessed + new**
- **HIGH — BLE duplicate device-picker** (#17, `connectivity.ts:987-1002`): the legacy `ble` node
  sets `bleDevices` *after* `await requestDevice()`, so a second frame opens a second chooser.
  Add a synchronous per-node creating-guard. (`bleDeviceExecutor`/`bleScannerExecutor` already safe.)
- **MED — clasp** (#13): `disconnect()` mutates **shared** `connection.config.autoReconnect=false`
  for all nodes on the id and never resets it (`clasp.ts:233`); the connect `Promise.race` timer is
  never cleared and a late-resolving client is never `close()`d (`:168-172`); `autoConnect ?? true`
  diverges from off-by-default (`:89-90`).
- **MED (NEW) — clasp video-receive `captureStream` + `<video>` never stopped** (`clasp.ts:968-981`
  vs `disposeClaspVideoReceiveNode:1440-1452`): the captured stream/tracks and `<video>` keep
  running on node removal.
- **MED (NEW) — clasp video-send `sendClient` double-connect** (`clasp.ts:1115-1120,1220`): the
  `broadcasting`/`encoder` guard is set *after* the `await builder.connect()`, so two frames each
  build a `sendClient` and one leaks. Set a synchronous `starting` guard before the await.
- **MED — `clearModelCache` doesn't cancel in-flight loads** (#16, `AIInference.ts:652-675`): a
  pending `loadModel` re-adds to `_loadedModels` after the clear. Guard with a `_cacheEpoch` counter.
- **Reassessed: #14 MediaPipe stream — NON-ISSUE as written.** `MediaPipeService` owns no stream;
  the webcam node in `visual.ts` creates and `track.stop()`s it correctly. (The real nearby leak is
  the clasp captureStream above.)
- **Reassessed: #15 STT `AudioBufferService` — already mitigated, LOW.** No `dispose()` but
  `disconnect()` does full teardown (stops tracks, closes AudioContext, clears buffers). Optional:
  alias `dispose = disconnect`. Note each STT node opens its own `AudioContext` (~6-context browser
  cap) — closed on disconnect.
- **LOW (NEW) — exported `audioBufferService` singleton (`AudioBufferService.ts:399`) appears
  unused** (every consumer constructs its own) — verify + remove the latent foot-gun.
