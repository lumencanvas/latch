# Deep Audit — Modernization Phases 0–2A (2026-06-14)

Adversarial review of the `modernization` branch (7 commits at audit time) before
implementing the Phase 2 dirty-flag engine. Method: re-read every changed file,
trace edge cases, run the full suite / typecheck / lint / web build, and read the
executors that the dirty-flag rewrite will classify. Findings below are split
into **fixed**, **latent (pre-existing) bugs found**, **design notes for Step B**,
and **outstanding/can't-verify-headless**.

- Unit suite **1126 passing / 11 todo (1137)**; typecheck 0 errors; eslint clean
  (only the expected `vite.config.ts` ignore warning).
- `npm run build:web` succeeds and ships `dist/web/coi-serviceworker.min.js` with

## Fixed this audit
- **Render-loop concurrency race (Phase 1).** `scheduleLoop()` re-armed
  `requestAnimationFrame` *after* `await executeFrame()` with no state recheck. If
  the tab was hidden (or stop/pause fired) during that async gap and then resumed,
  a stale loop and the new loop could both run → frames at 2×. Fixed with a
  monotonic `loopToken`: a loop re-arms only while it's still active, and
  stop/pause/hidden call `invalidateLoop()`. Regression test added (fails on the
  pre-fix code). Committed in `b2c589d`.

## Verified safe (claims I'd previously asserted)
- **Electron is unaffected by the web service worker.** `electron.vite.config.ts`
  builds the renderer from `src/renderer/index.html` and `src/main/index.ts` does
  `loadFile('../renderer/index.html')`. The coi-serviceworker `<script>` lives only
  in the *root* `index.html` (web/Vite entry), so there's no SW reload-loop risk in
  the desktop app. It also self-gates on `isSecureContext` + `crossOriginIsolated`.
- **COEP `credentialless` is the correct choice.** The app loads cross-origin,
  no-cors subresources (Google Fonts CSS/woff2 via `<link>`, HF model weights and
  MediaPipe WASM via fetch). `require-corp` would block those unless each sends
  CORP; `credentialless` lets them load (credentials stripped) while still enabling
  `crossOriginIsolated`. Safari (no credentialless) degrades to non-isolated /
  single-threaded — still functional.

## Latent (pre-existing) bugs found — NOT introduced by this work
- **`smooth` node is a silent no-op.** `smoothExecutor` reads `_prev` from controls
  and writes `_prev` as an *output*, but the engine's `handleSpecialOutputs` never
  feeds `_prev` back into controls (it only handles `_dynamicInputs/_dynamicControls/`
  `_dynamicOutputs/_preset_code/_controlUpdates`). So `prev` is always `undefined`,
  `prev ?? target = target`, and `smoothed = target + (target-target)*… = target`
  every frame — i.e. **smoothing does nothing; output == input**, regardless of the
  `factor`/`deltaTime`. The code comment even says "this would need special handling
  in the engine." Fix later by routing `_prev` through `_controlUpdates`, or persist
  state in a module Map like the other stateful nodes. (Consequence: golden flow #5
  "stateful smoothing converges" is mislabeled — it currently just passes through.)

## Design notes carried into Phase 2 Step B
- **Purity must be verified by reading, not grepping.** Grepping executors for
  `getState`/`Math.random`/`ctx.totalTime`/etc. is insufficient: `gate` has zero
  such markers yet is stateful via a module-level `gateLastValue` Map. Therefore
  `data.ts`/`string.ts` showing "0 markers" does **not** prove purity. The
  dirty-flag pure-set is built **only from directly-read executors**.
- **Verified PURE node types (safe to skip when inputs+controls unchanged):**
  `constant, add, subtract, multiply, divide, map-range, clamp, abs, trig, power,
  vector-math, modulo, lerp, step, smoothstep, remap, quantize, wrap, compare,
  and, or, not, select, switch` (24 types). Everything else defaults to
  **always-run** (safe-by-default — a misclassification can never freeze a node;
  worst case is no speedup). Notable exclusions and why: `gate` (module state),
  `smooth` (uses `ctx.deltaTime`), `random` (`Math.random`), `start`/`trigger`/
  `interval`/`delay`/`timer`/`metronome`/`step-sequencer`/`console`/`monitor`/
  `oscilloscope`/`equalizer` (module-level state), `time`/`lfo` (time-dependent).
- **Equivalence-harness state leak.** Executor state is module-global keyed by
  `nodeId`. Running the same flow full-then-dirty would let the first run's state
  bleed into the second for module-stateful nodes. Mitigation: the equivalence
  harness namespaces node IDs per run (distinct prefixes) so state can't collide.
  (The current golden flows happen to use no module-stateful nodes, but the
  equivalence harness should not depend on that.)
- **Dirty mode must reuse `executeNode`** (which also runs `handleSpecialOutputs`
  + metrics) so executed nodes behave identically to full mode; only the *decision
  to skip* is new. Change detection = `firstRun || controlsChanged ||
  anyUpstreamChangedThisFrame`. "Changed" is computed conservatively: primitives by
  `===`; any object/array output ⇒ treated as changed (never wrongly skipped).

## Minor / accepted
- `nodeById` Map keeps the *last* entry on duplicate node IDs, vs `Array.find`'s
  *first*. Duplicate IDs are invalid in Vue Flow; accepted.
- `start()` while status is `paused` schedules a loop that immediately returns
  (store stays paused). Pre-existing guard behavior; the real path is `toggle()` →
  `resume()`. Harmless; accepted.
